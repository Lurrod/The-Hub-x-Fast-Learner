# Rating 2.0 — Valorant-adapted HLTV 2.0 stats system

**Status:** Draft — pending implementation
**Date:** 2026-06-01
**Owner:** main bot
**Reference:** [HLTV Rating 2.0 article](https://www.hltv.org/news/20695/introducing-rating-20)

## Goal

Refactor the current `/stats` command (ELO-only) into a full per-queue
performance dashboard backed by a HLTV 2.0-equivalent rating adapted
for Valorant. Stats are computed from Henrik match data at match
verification time and persisted in two collections for O(1) reads.

## Scope decisions (locked)

- **Rating depth:** HLTV 2.0 faithful — parse Henrik `rounds[]` array
  for KAST, multikills, opening duels, traded kills.
- **Backfill:** Forward-only. Matches verified before this ships keep
  their ELO/W/L only; new matches start populating the new collections.
- **Segmentation:** Per-queue only (`pro` / `semipro` / `open` / `gc`),
  same axis as the existing ELO collection. No global, no per-map.
- **Display:** Two-page paginated embed (overview + impact details).
- **Min games:** None. Rating is computed from game 1; "(N games)"
  footer flags low-sample volatility.

## Architecture

### Section 1 — Henrik data extraction (`services/riot_api.py`)

The current `MatchPlayerStats` only carries `kills/deaths/assists/score/
agent/team`. Extend it with everything Rating 2.0 needs:

```python
@dataclass(frozen=True)
class MatchPlayerStats:
    puuid: str
    name: str
    tag: str
    team: str
    score: int          # combat score (ACS source)
    kills: int
    deaths: int
    assists: int
    agent: str
    # NEW
    damage_made: int
    damage_received: int
    headshots: int
    bodyshots: int
    legshots: int
    multikills_2k: int
    multikills_3k: int
    multikills_4k: int
    multikills_5k: int
    first_kills: int
    first_deaths: int
    kast_rounds: int     # count of rounds with K/A/S/T
```

Parser changes in `_parse_henrik_match`:

1. Aggregate stats from `player.stats` (damage_made, headshots, etc.)
   already present in the v3 payload.
2. Iterate `data.rounds[]` once. For each round, for each player:
   - `kills_in_round = len([e for e in round.player_stats[player].kill_events])`
     (or via the round-level `kill_events`)
   - Multikills: increment the appropriate `multikills_Nk` counter
     if `kills_in_round in {2,3,4,5}`.
   - Survived: True iff the player is not in the victim list.
   - Traded: True iff this player died and a teammate killed the
     attacker within 5s of the player's death (use `kill_events`
     timestamps).
3. First kill / first death: in each round's `kill_events`, the
   earliest event by `kill_time_in_round` defines FK (killer) and
   FD (victim).
4. KAST round: player counts for KAST in round R iff they had a
   kill, an assist, survived, OR died-but-traded.

If the rounds array is absent or malformed for a given match (Henrik
hiccup), parsing falls back gracefully: per-round counters default to
zero, and the match is still persisted with the aggregate-only stats
(kills/deaths/assists/damage). A `parse_warnings` field on the doc
flags this so we can audit.

### Section 2 — Rating formula (`services/rating.py`, new module)

Faithful HLTV 2.0 port:

```
Rating 2.0 = 0.0073 * KAST          // KAST as percentage (0-100)
           + 0.3591 * KPR
           - 0.5329 * DPR
           + 0.2372 * Impact
           + 0.0032 * ADR
           + 0.1587

Impact     = 2.13 * KPR + 0.42 * APR - 0.41
```

With Valorant inputs:
- `KAST = (kast_rounds / rounds_played) * 100`
- `KPR  = kills / rounds_played`
- `DPR  = deaths / rounds_played`
- `APR  = assists / rounds_played`
- `ADR  = damage_made / rounds_played`

API:

```python
@dataclass(frozen=True)
class RatingInputs:
    rounds_played: int
    kills: int
    deaths: int
    assists: int
    damage_made: int
    kast_rounds: int

def compute_rating_2_0(inputs: RatingInputs) -> float:
    if inputs.rounds_played <= 0:
        return 0.0
    kast = (inputs.kast_rounds / inputs.rounds_played) * 100
    kpr  = inputs.kills / inputs.rounds_played
    dpr  = inputs.deaths / inputs.rounds_played
    apr  = inputs.assists / inputs.rounds_played
    adr  = inputs.damage_made / inputs.rounds_played
    impact = 2.13 * kpr + 0.42 * apr - 0.41
    return (
        0.0073 * kast
        + 0.3591 * kpr
        - 0.5329 * dpr
        + 0.2372 * impact
        + 0.0032 * adr
        + 0.1587
    )
```

The aggregate doc recomputes the rating from its summed counters on
every read, so the formula constants can be re-tuned in code without
a backfill.

Two new collections.

**`match_player_stats`** — 1 doc per (match, player):

```json
{
  "_id": "{match_id}:{user_id}",
  "match_id": ObjectId,
  "user_id": "123",
  "queue_type": "pro",
  "matchid_henrik": "abc-def",
  "map": "Ascent",
  "agent": "Jett",
  "rounds_played": 24,
  "win": true,
  "kills": 22, "deaths": 14, "assists": 5,
  "damage_made": 4123, "damage_received": 3580,
  "headshots": 18, "bodyshots": 50, "legshots": 4,
  "multikills_2k": 3, "multikills_3k": 1,
  "multikills_4k": 0, "multikills_5k": 0,
  "first_kills": 4, "first_deaths": 2,
  "kast_rounds": 19,
  "acs": 287,
  "rating_2_0": 1.34,
  "created_at": ISODate,
  "parse_warnings": []

```

Indexes: PK is the compound `_id`. Secondary index
`(user_id, queue_type, created_at desc)` for future
`/match-history` paging.

**`player_rating_aggregates`** — 1 doc per (player, queue):

```json
{
  "_id": "123:pro",
  "user_id": "123",
  "queue_type": "pro",
  "games": 47,
  "rounds_played": 1128,
  "kills": 1034, "deaths": 891, "assists": 312,
  "damage_made": 196443, "damage_received": 174022,
  "headshots": 781, "bodyshots": 2150, "legshots": 188,
  "multikills_2k": 142, "multikills_3k": 28,
  "multikills_4k": 7,   "multikills_5k": 1,
  "first_kills": 198, "first_deaths": 156,
  "kast_rounds": 856,
  "rating_2_0_sum": 62.9,
  "updated_at": ISODate
}
```

Note: `rating_2_0_sum` is the simple sum of per-match ratings — not
used to display the rating (we recompute from raw counters above) but
kept for future "average per-match rating" display variants.

New repository helpers:
- `insert_match_player_stats(db, docs: list[dict]) -> int`
  Bulk insert with `ordered=False`, swallow duplicate-key errors (idempotent
  retry). Returns inserted count.
- `update_rating_aggregates(db, deltas: list[dict]) -> None`
  For each delta, `find_one_and_update({_id}, {$inc: {...}, $set: {updated_at}}, upsert=True)`.
- `get_rating_aggregate(db, *, user_id: int, queue_type: str) -> Mapping | None`
- `get_player_match_stats(db, *, user_id: int, queue_type: str, limit: int) -> list`

### Section 4 — Verify pipeline integration

`services/match_verifier.py`:

```python
@dataclass(frozen=True)
class PlayerStatsExtended:
    # PlayerPerformance fields ...
    user_id: str
    puuid: str
    acs: float
    multiplier: float
    win: bool
    # NEW
    queue_type: str
    map_name: str
    agent: str
    rounds_played: int
    kills: int
    deaths: int
    assists: int
    damage_made: int
    damage_received: int
    headshots: int
    bodyshots: int
    legshots: int
    multikills_2k: int
    multikills_3k: int
    multikills_4k: int
    multikills_5k: int
    first_kills: int
    first_deaths: int
    kast_rounds: int
    rating_2_0: float

def build_extended_stats(
    match: MatchSummary,
    *,
    puuid_to_user_id: Mapping[str, str],
    queue_type: str,
) -> tuple[PlayerStatsExtended, ...]: ...
```

`cogs/match/_cog.py` — after Henrik verification claims ELO, but in
the same logical step:

1. `extended = build_extended_stats(match, puuid_to_user_id=..., queue_type=match_doc["queue_type"])`
2. `match_stats_docs = [_to_match_player_stats_doc(s, match_id) for s in extended]`
3. `inserted = await asyncio.to_thread(repository.insert_match_player_stats, db, match_stats_docs)`
   - If `inserted == 0` (all duplicates), skip the aggregate update — we
     have already processed this match. This is the idempotence gate.
4. `deltas = [_to_aggregate_delta(s) for s in extended]`
5. `await asyncio.to_thread(repository.update_rating_aggregates, db, deltas)`

Failure isolation: a Mongo error in step 3 or 5 must log + warn but
not undo the ELO application. We treat stats persistence as
best-effort secondary; the ELO is the system of record.

**Pagination view** (`cogs/stats/_view.py`):

```python
class StatsPaginatorView(discord.ui.View):
    """Two-page paginator. Stores the two pre-built embeds + the
    invoker_id; flips `page` on button press and re-edits the
    interaction message. timeout=300s; on timeout, disable both
    buttons in-place (no message edit if already gone).
    """

    def __init__(self, *, overview, details, invoker_id: int):
        super().__init__(timeout=300)
        self.overview = overview
        self.details = details
        self.invoker_id = invoker_id
        self.page = 0
```

Implementation rules (called out so plan/impl phase can't drift):
- Two `@discord.ui.button` decorated methods `prev` / `next` with
  stable `custom_id`s `"stats_prev"` / `"stats_next"`. Each method
  flips `self.page` (0 ↔ 1), then awaits
  `interaction.response.edit_message(embed=<current>, view=self)`.
- `interaction_check` returns False (ephemeral "Not your stats panel")
  if `interaction.user.id != self.invoker_id` — prevents two viewers
  fighting over the buttons.
- Stable `custom_id`s mean the view can be re-registered at startup
  via `bot.add_view(StatsPaginatorView(...))` with placeholder embeds,
  so clicks on stale messages still get routed (graceful "session
  expired" reply rather than a dead click).

### Section 7 — Migration & rollout

- No DB migration: new collections appear naturally at first use.
- `cogs/elo_admin.py:/stats` is removed (or kept as a stub that points
  to the new command for one release if we want a deprecation window —
  default: hard cut, since slash command names don't collide).
- Indexes are created lazily by the existing `_ensure_indexes` pattern
  in `repository.py`.

## Non-goals

- Map-level breakdown (Ascent/Bind/etc. per-map rating).
- Cross-queue aggregate ("Global rating").
- Historical backfill of pre-deployment matches.
- Leaderboard sorted by Rating 2.0 (the current leaderboard stays
  ELO-sorted; a separate `/leaderboard-rating` is out of scope here).

## Risks

- **Henrik rounds payload variability**: not every match guarantees
  consistent `kill_events` ordering. Mitigation: parse defensively,
  default to zero on missing keys, flag `parse_warnings`.
- **Multi-PUUID players (smurf accounts)**: stats key on `user_id`
  resolved from PUUID → linked Discord. If a player switches Riot
  accounts mid-history, the aggregate jumps. Acceptable: ELO has the
  same property today.
- **Idempotence on Henrik retries**: the verifier retries on
  transient failures. The compound `_id = {match_id}:{user_id}` and
  the "skip aggregate update if insert returns zero" guard prevent
  double-counting.
